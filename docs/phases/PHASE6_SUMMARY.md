# Phase 6 Summary — Planning Layer

## Goal

Phase 5 built a reliable execution pipeline: Story → code → tests → PR → merge.

Phase 6 added the layer above it: the ability to take a large requirement (an Epic) and decompose it into structured, reviewable, executable Stories — with a mandatory human approval gate before anything touches Jira.

---

## What Was Built

### End-to-end planning flow

```
Jira Epic → "Ready for Breakdown"
  → Dispatcher routes to epic_breakdown workflow
  → Worker calls Claude (structured tool output)
  → Proposals stored in DB as PROPOSED
  → Telegram: breakdown sent for review with APPROVE / REJECT / REGENERATE
  → Run enters WAITING_FOR_APPROVAL

User replies via Telegram or HTTP:

  APPROVE → Stories created in Jira (sequentially, each key persisted)
            → run transitions COMPLETED
            → Telegram: list of created keys + titles

  REJECT  → run transitions FAILED/REJECTED
            → proposals marked REJECTED
            → no Jira issues created

  REGENERATE → old run closed (FAILED/REGENERATE_REQUESTED)
               → new run queued, Claude called again
               → fresh proposals sent for review

Created Stories → status "Ready for Dev"
  → existing story_implementation pipeline picks them up automatically
```

## New Components

### `app/claude_client.py` additions

- `plan_epic_breakdown(issue_key, summary) -> dict`
- Uses `claude-sonnet-4-6`, `max_tokens=4096`
- Tool-forced output (`tool_choice={"type": "tool", "name": "plan_breakdown"}`)
- Schema: `items[]` (title, description, acceptance_criteria, rationale, dependency_notes, risk_notes, confidence) + `summary`, `assumptions`, `open_questions`
- Caps output at `MAX_STORIES_PER_EPIC = 8`
- Raises `RuntimeError` on empty or malformed output

### `app/jira_client.py` (new file)

- `create_story_under_epic(project_key, epic_key, title, run_id, description, acceptance_criteria, rationale, dependency_notes, risk_notes) -> str`
- Builds Atlassian Document Format (ADF) body with all fields
- Includes traceability footer: `Generated by AI Dev Orchestrator | Planning run: {run_id} | Parent Epic: {epic_key}`
- Adds `ai-generated` label to every created Story
- Uses `parent` field for Epic → Story linking (Jira Cloud next-gen)

### `app/telegram.py` additions

- `parse_approval_command(text) -> tuple[str, int] | None`
- `set_webhook(webhook_url) -> dict`

### `app/workflows.py` additions

- `epic_breakdown(run_id, issue_key, issue_type, summary)` — full planning workflow
- `create_jira_stories_for_run(run_id, issue_key)` — called inline from Telegram APPROVE handler

### `app/webhooks.py` additions

- `POST /webhooks/telegram` — receives and routes Telegram approval commands
- APPROVE/REJECT use strict `get_pending_planning_run` lookup
- REGENERATE uses broader `get_planning_run_for_regeneration` (accepts pending and completed runs)

## Data Model Changes

### `workflow_runs` — new columns

| Column | Type | Purpose |
|---|---|---|
| `parent_issue_key` | VARCHAR(100) | Epic key this planning run is decomposing |
| `approval_status` | VARCHAR(30) | PENDING → APPROVED / REJECTED / REGENERATE_REQUESTED |
| `approval_requested_at` | TIMESTAMP | When proposals were sent to Telegram |
| `approval_received_at` | TIMESTAMP | When user replied |
| `created_jira_children_count` | INTEGER | Count of Stories actually created |
| `assumptions_json` | TEXT | JSON array of Claude's stated assumptions |
| `open_questions_json` | TEXT | JSON array of Claude's open questions |

### `planning_outputs` — new table

Stores each proposed Story individually. One row per item per run.

| Column | Purpose |
|---|---|
| `run_id` | FK to workflow_runs |
| `parent_issue_key` | Epic key |
| `sequence_number` | Position in breakdown (1–8) |
| `title` | Story summary |
| `description` | Implementation-oriented description |
| `acceptance_criteria` | Bulleted list as plain text |
| `rationale` | Why this Story exists |
| `dependency_notes` | Ordering hints |
| `risk_notes` | Known risks |
| `confidence` | high / medium / low |
| `status` | PROPOSED → CREATED / REJECTED |
| `created_issue_key` | Jira key once created (e.g. KAN-24) |

### `workflow_runs` status additions

```
QUEUED → RUNNING → WAITING_FOR_APPROVAL → APPROVED → COMPLETED
                                        ↘ FAILED (REJECTED / REGENERATE_REQUESTED)
```

## Key Design Decisions

**Epic → Story only (no Feature level)**
The default Jira hierarchy (Epic → Story) was used. Features and Tasks are not modelled. Stories are the atomic unit of implementation.

**Approval is mandatory**
Planning output is a proposal. Claude never creates Jira issues directly. Every breakdown requires an explicit APPROVE before any Jira API call is made.

**Inline Jira creation**
Story creation happens synchronously inside the Telegram webhook handler, not in a separate worker job. Rationale: 8 Jira API calls at ~300ms each is well within Telegram's 60-second webhook timeout, and avoids adding a new job type.

**Idempotency via `planning_outputs.status`**
Before running Claude, the worker checks for any `CREATED` planning_outputs for the same parent Epic (excluding the current run). If found, the run is immediately failed with a `planning_duplicate_blocked` message pointing to the existing run. REGENERATE marks prior outputs as REJECTED before the new run starts, so the check passes cleanly.

**REGENERATE scope**
`get_planning_run_for_regeneration` accepts both `WAITING_FOR_APPROVAL+PENDING` (mid-review) and `COMPLETED+APPROVED` (after children were already created). This allows users to discard an approved breakdown and start fresh without needing a new Jira webhook event.

**Traceability**
Every created Jira Story carries: `ai-generated` label, ADF description with rationale/dependencies/risks sections, and an italic footer with planning run ID and parent Epic key. This makes every AI-generated issue auditable without needing DB access.

## Telegram Commands

| Command | Effect |
|---|---|
| `APPROVE <run_id>` | Accept proposals, create Stories in Jira |
| `REJECT <run_id>` | Discard proposals, no Jira issues created |
| `REGENERATE <run_id>` | Close current run, re-run Claude, send new proposals |

Chat ID filtering: messages from any chat other than `TELEGRAM_CHAT_ID` are silently ignored.

---

## Telegram Event Types Added

| Event | When |
|---|---|
| `epic_breakdown_started` | Workflow begins |
| `epic_breakdown_proposed` | Proposals ready, awaiting approval (includes titles, confidence, assumptions, open questions) |
| `epic_breakdown_approved` | User approved, Jira creation in progress |
| `epic_breakdown_complete` | All Stories created (lists key + title for each) |
| `epic_breakdown_rejected` | User rejected, nothing created |
| `epic_breakdown_regenerate` | Old run closed, new run queued |
| `epic_breakdown_failed` | Unrecoverable error |
| `planning_duplicate_blocked` | Automatic re-trigger blocked, REGENERATE hint included |
| `approval_error` | Command received for a run that is not in a valid approval state |

## What Phase 6 Did Not Build (by design)

- `feature_breakdown` workflow (no Feature level in this Jira setup)
- Autonomous creation without approval
- Sprint planning or reprioritisation
- Multi-agent planning
- Memory engine or external knowledge ingestion
- Automatic Story execution after creation (Stories require manual status move to trigger execution)
- Automatic closure of parent Epics
